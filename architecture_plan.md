# Локальный AI-координатор сервис-деска — архитектура и план

## 1. Что строим

Система на ноутбуке, с веб-интерфейсом, которая:
- слушает корпоративный Discord;
- читает корп. почту (Gmail);
- собирает данные из самописного трекера задач (без API → веб-скрейпинг);
- работает с трекером сервисных заявок;
- индексирует Google Drive;
- индексирует вики/документацию в Notion;
- парсит фото (скрины, фото оборудования, документы) в базу знаний;
- выполняет роль координатора сервис-деска — принимает заявки, классифицирует, назначает, отвечает, эскалирует;
- вся ИИ-обработка идёт через Router AI (например OpenRouter) — без локальных моделей.

## 2. Почему custom Python, а не n8n

Почти каждый источник данных требует нетривиальной логики: скрейпинг трекера без API, Discord-бот с сохранением истории, агент-координатор с function calling, парсинг фото. Готовых нод под это в n8n нет — низкий код в такой конфигурации не экономит время, а добавляет слой абстракции. Поэтому — модульный backend на Python (FastAPI), с общей БД и очередью задач. n8n можно добавить позже как визуальный слой для простых вспомогательных сценариев (уведомления, отчёты), но не как ядро.

## 3. Архитектура (модули)

```
                          ┌─────────────────────┐
                          │      Web UI          │  (дашборд, чат с координатором,
                          │  React/Next.js        │   поиск по базе знаний, тикет-борд)
                          └──────────┬───────────┘
                                     │ REST/WebSocket
                          ┌──────────▼───────────┐
                          │   API Gateway (FastAPI)│
                          └──────────┬───────────┘
                                     │
        ┌───────────┬───────────┬─────────┬─────────────┬─────────────┐
        ▼           ▼           ▼         ▼             ▼             ▼
   ┌─────────┐ ┌─────────┐ ┌────────┐ ┌────────┐  ┌────────────┐┌───────────┐
   │ Discord │ │ Gmail    │ │ Drive  │ │ Notion │  │ Tracker    ││ Service   │
   │connector│ │connector │ │connector│ │connector│ │ scraper    ││ desk conn.│
   └────┬────┘ └────┬────┘ └───┬────┘ └───┬────┘  └──────┬─────┘└─────┬─────┘
        │           │          │          │              │            │
        └───────────┴────┬─────┴──────────┴──────────────┴────────────┘
                               ▼
                     ┌───────────────────┐
                     │  Celery workers    │  (нормализация, чанкинг,
                     │  + Redis (очередь) │   эмбеддинги, OCR/vision)
                     └─────────┬─────────┘
                               ▼
                     ┌───────────────────┐
                     │  Router AI клиент  │  (единая точка выхода в LLM/vision/embeddings)
                     └─────────┬─────────┘
                               ▼
                ┌──────────────────────────────┐
                │ PostgreSQL + pgvector          │  (структурные данные + база знаний/RAG)
                │ + локальное файловое хранилище │  (сырые файлы, фото, вложения)
                └──────────────────────────────┘
                               ▲
                     ┌─────────┴─────────┐
                     │ Coordinator Agent  │  (агентный цикл с tool-calling:
                     │                    │   создать тикет, найти в KB, ответить
                     │                    │   в Discord/почте, эскалировать)
                     └───────────────────┘
```

Каждый коннектор — независимый процесс/контейнер, который только пишет нормализованные данные в общую БД и очередь. Это даёт модульность: можно выключить/переписать любой источник без остановки остальных.

## 4. Стек по компонентам

| Компонент | Технология | Почему |
|---|---|---|
| Backend/API | Python 3.12 + FastAPI | async, богатая экосистема под все интеграции |
| БД | PostgreSQL + расширение pgvector | структура и вектора в одном месте, меньше операционной сложности, чем отдельный Qdrant |
| Очередь/фон. задачи | Redis + Celery | надёжные ретраи для скрейпинга/эмбеддингов, гибкое планирование |
| Файлы/фото | локальная папка (по модулям и датам), опционально MinIO позже | ноутбук — S3-совместимость пока не нужна |
| Discord | discord.py (бот, read + message content intent) | стандарт де-факто, стабильная библиотека |
| Gmail | Gmail API (google-api-python-client), OAuth2 | официальный путь, инкрементальная синхронизация через History API |
| Google Drive | Drive API v3 + changes.list | инкрементальная индексация без полного пересканирования |
| Notion | официальный Notion API (integration token) | вики/база знаний — есть нормальный API, скрейпинг не нужен |
| Трекер без API | Playwright (headless Chromium, Python) | справляется с JS-интерфейсами, держит сессию/логин |
| Сервис-деск заявки | то же (API если найдётся, иначе тот же Playwright-подход) | уточнить отдельно, какая система используется |
| Парсинг фото | vision-модель через Router AI (например gpt-4o/gemini/claude vision) | не нужна локальная GPU-модель, качество выше |
| Эмбеддинги/RAG | embeddings через Router AI + pgvector + простой retrieval (LangChain или свой) | минимум зависимостей, легко заменить модель |
| Координатор | агентный цикл с function-calling через Router AI | конфигурируемые "инструменты": create_ticket, search_kb, reply, escalate |
| Web UI | Next.js + Tailwind (или проще — FastAPI + HTMX, если не нужна тяжёлая интерактивность) | React — если нужен богатый UI, HTMX — если важна простота сборки на одном ноутбуке |
| Развёртывание | Docker Compose (все модули контейнерами) | воспроизводимо, легко включать/выключать модули |
| Доступ извне ноутбука | Tailscale (если нужен доступ не только с этого устройства) + basic auth на UI | ноутбук не в публичной сети |

## 5. Router AI — единая точка выхода в модели

Отдельный модуль `router_ai_client`, который:
- принимает тип задачи (классификация, суммаризация, диалог координатора, vision/OCR, эмбеддинги);
- выбирает модель под задачу через конфиг (yaml/env), не хардкодя модель в коде;
- логирует затраты/токены;
- даёт единый интерфейс остальным модулям — при смене провайдера/модели меняется только конфиг.

Важно: почта, чаты и фото — корпоративные данные, они пойдут во внешний Router AI. Стоит заранее (не в коде, а организационно) проверить с юристами/ИБ, что это допустимо, и по возможности исключать явно чувствительные каналы/переписку из автоматической обработки (например, через список исключений по каналам/меткам).

## 7. Дорожная карта (по фазам, каждая — рабочий инкремент)

1. **Инфраструктура**: Docker Compose (Postgres+pgvector, Redis, FastAPI-скелет, пустой веб-дашборд), базовая авторизация.
2. **База знаний из Google Drive + Notion**: индексация файлов и вики-страниц, чанкинг, эмбеддинги, поиск в UI. Notion добавляется первым в этой фазе — простой API, минимум сложности, максимум ценности сразу.
3. **Discord-коннектор**: чтение сообщений, сохранение, попадание в базу знаний + простая классификация (вопрос/инцидент/шум).
4. **Gmail-коннектор**: аналогично, с фильтрацией по меткам/папкам.
5. **Скрейпер трекера задач**: самый хрупкий модуль — изолировать, добавить алерты на случай изменения интерфейса трекера.
6. **Сервис-деск + координатор-агент**: тикет-логика, приоритизация, назначение, автоответы, эскалация человеку. Зависит от пунктов 2–5.
7. **Парсинг фото**: vision-пайплайн → структурированные записи в базе знаний.
8. **UI-полировка**: тикет-борд, дашборд, чат с координатором, история.
9. **Хардненинг**: логи/аудит, ретраи, редактирование чувствительных данных перед отправкой в Router AI, резервное копирование БД.

## 8. Риски, на которые стоит смотреть заранее

- Скрейпер трекера ломается при любом обновлении его UI — нужны алерты и быстрый фикс селекторов.
- Discord-боту нужен admin-доступ на сервере + включённый Message Content Intent.
- Gmail/Drive потребуют OAuth consent screen, а для доступа не только к своему ящику — одобрение админа Google Workspace (domain-wide delegation).
- Отправка корп. данных во внешний Router AI — вопрос соответствия политике компании, решить до продакшена, не после.

## 9. Что нужно от тебя перед стартом реализации

- Название/тип сервис-деск системы для заявок (чтобы понять, есть ли у неё API, в отличие от трекера задач).
- Права: админ Discord-сервера, Google Workspace admin (или обычный OAuth на свой ящик), доступ/логин к трекеру для скрейпера.
- Аккаунт в Router AI (OpenRouter или аналог) и бюджет на токены.
