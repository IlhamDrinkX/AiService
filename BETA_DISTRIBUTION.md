# Раздача бета-тестерам через GitHub

## Что уже готово

- `install.bat` — ставит Git (если его нет), клонирует репозиторий (ветку `main`).
- `setup_all.bat` — ставит venv + зависимости для всех модулей одним прогоном,
  запускает веб-морду.

## Что нужно сделать тебе (не автоматизировано — требует твоего GitHub-аккаунта)

1. Создать репозиторий на GitHub (например `AI_service_system`) — публичный
   или приватный, смотря кому будешь давать доступ.
2. Запушить туда код: `git remote add origin <URL>` → `git push -u origin master:main`.
   Локальный git-репозиторий уже инициализирован, `.gitignore` уже настроен
   и исключает секреты/токены/локальные данные — можно коммитить как есть.
3. В `install.bat` поправить строку `set REPO_URL=...` на настоящий адрес.
4. Раздать `install.bat` бета-тестерам (просто файл, без остального кода —
   он сам всё скачает).

## Про "приложение должно использовать мои токены авторизации"

Это прямая просьба, но у неё есть реальный риск: если зашить твои личные
`OPENROUTER_API_KEY`/Google OAuth client secret в файлы, которые лежат в
GitHub-репозитории — а тестировать будут несколько человек — эти ключи
утекут наружу вместе с репозиторием (даже приватным: любой, у кого есть
доступ к репо, увидит их в истории коммитов навсегда, `git rm` это не
отменяет). Дальше это либо чужой доступ к твоей личной почте/Google Drive
(OAuth-токен), либо чужой счёт на твоей карте (OpenRouter — платный API).

Вместо этого сделано так же, как для обычных пользователей
(`SETUP_NEW_USER.md`) — `.env`/`credentials.json` никогда не попадают в git
— но чтобы бета-тестерам не пришлось заводить *свои* ключи с нуля, есть
компромисс:

- `setup_all.bat` умеет подхватить папку `beta_credentials/` (если она лежит
  рядом с самим батником на машине тестера) — и разложить из неё готовые
  `.env`/`credentials.json` по всем нужным местам поверх `.env.example`.
- Эта папка **никогда не попадает в git** (`.gitignore` это гарантирует) —
  раздаёшь её тестерам отдельно, не через GitHub: например архивом по
  прямой ссылке в личном чате, а не публикуешь.
- Так владеешь контролем, кому именно она досталась, и можешь отозвать
  доступ (перевыпустить ключ OpenRouter, отозвать OAuth-токен в Google
  Account → Security → Third-party access), если тестирование закончилось
  или кто-то один поделился архивом дальше не тем, кем следовало.

Если это осознанный компромисс, с которым ты согласен (например, бета —
несколько доверенных коллег, а не публичный релиз) — собери
`beta_credentials/` из своих текущих `.env`/`credentials.json`/`token.json`
файлов (списком путей — см. `.gitignore`, там же перечислены все паттерны,
которые не в git) и раздай отдельно. Если нет — каждый тестер заводит свои
ключи по `SETUP_NEW_USER.md`, дольше, но безопаснее.

## Discord-бот отдельно

`connectors/discord/` — это один живой бот на сервере DrinkX (Discord).
Если бета-тестеры не должны видеть переписку DrinkX — у бота нет смысла
шариться, каждый тестер либо не подключает Discord-модуль вообще, либо
заводит своего бота на своём тестовом Discord-сервере (`SETUP_NEW_USER.md`,
раздел Discord).
