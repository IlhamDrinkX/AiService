# reporting — сводные отчёты по заявкам/тикетам/задачам

MVP модуля 1 из `functional_plan_ui.md` (лента задач) — часть, которую попросили
первой: сводка по периоду с выводами, обязательно со ссылками на тикеты/задачи,
на которых основаны выводы. Вывод — текст в терминал (как чат-ответ) или
самодостаточный HTML-файл для открытия в браузере.

## Источники

Читает SQLite всех трёх коннекторов заявок напрямую (`connectors/*/data/*.db`) —
не импортирует сами коннекторы, у них разные несовместимые зависимости в
отдельных venv (curl_cffi, playwright), а тут нужен только `SELECT`:

- Service Desk — `servicedesk_tickets`
- Fibbee ERP — `fibbee_tickets`
- Tracker — `tracker_tasks`

Если по какому-то источнику ещё не запускался `sync.py` — соответствующая база
пуста или отсутствует, генератор просто пропустит источник (не упадёт).

## ⚠️ Ссылки на источники — нужно проверить

`link_templates.yaml` — адреса карточек заявок в вебе, но они **не подтверждены
в браузере** (в отличие от самих API-путей в клиентах коннекторов, которые
проверены живыми запросами). Открой любую заявку/тикет/задачу в браузере и
сверь адресную строку с шаблоном — если не совпадает, поправь `link_templates.yaml`.

## Запуск

Отдельного venv нет — использует зависимости `core/router_ai` (requests,
python-dotenv, PyYAML — их достаточно).

```
core\router_ai\venv\Scripts\python core\reporting\generate.py --source all --days 7 --format html
```

Опции:
- `--source {all,servicedesk,fibbee,tracker}` — по умолчанию all.
- `--days N` — глубина периода (0 = без ограничения). Дробные значения ок (0.5 = 12 часов).
- `--question "..."` — свой вопрос модели; по умолчанию общая сводка (статистика по
  статусам, повторяющиеся проблемы, что требует внимания).
- `--format {text,html}` — text печатает в терминал с markdown-ссылками, html
  сохраняет файл в `reports/` (в git не попадает — реальные операционные данные).
- `--tier {nano,flash,pro}` — уровень модели, по умолчанию flash. Для быстрой
  проверки на дешёвом уровне — `--tier nano`.
- `--max-items N` — жёсткий потолок числа заявок в одном вызове модели (по
  умолчанию 60) — защита от неожиданно большого счёта на широких периодах.

## Как это работает

1. Данные из трёх баз собираются в единый список с полями `ref`/`source`/`title`/
   `status`/`url`/`text`, сортируются по дате, обрезаются до `--max-items`.
2. `RouterAIClient.report()` (в `core/router_ai/client.py`) строит пронумерованный
   список источников и требует от модели ссылаться на них как `[N]`, без
   выдумывания фактов вне списка.
3. `generate.py` заменяет `[N]` на настоящие markdown/HTML-ссылки — это делается
   здесь, а не в промпте, чтобы не тратить токены на сами URL и не рисковать, что
   модель их исказит.

## Дальше по плану

Пока читает по всем заявкам без учёта группировки по темам/кластерам из модуля 1
(`functional_plan_ui.md`, §1.1) — она появится, когда будет общий FastAPI
read-layer с уже посчитанными темами. До этого отчёт группирует данные только по
источнику и статусу, как попросит вопрос модели.
