# router_ai — единая точка выхода в LLM (OpenRouter + прямой Gemini)

Один клиент (`client.py`), через который проходят все вызовы моделей: классификация
задач, суммаризация Discord, ответы базы знаний, сводные отчёты. Ни один другой модуль
не должен ходить в OpenRouter или в Gemini напрямую — иначе смена модели/уровня/провайдера
превращается в правку кода по всему проекту вместо правки одного `models.yaml`.

Подробности про сами три уровня и почему так — в `../../functional_plan_ui.md`, §4.

## Два провайдера

Каждый уровень (`nano`/`flash`/`pro`) в `models.yaml` может стоять на одном из двух
провайдеров (поле `provider`, по умолчанию `openrouter`):

- **`openrouter`** — платно, любая модель каталога OpenRouter. Нужен `OPENROUTER_API_KEY`.
- **`gemini_direct`** — прямой вызов Google Generative Language API по корпоративному
  ключу (`GEMINI_API_KEY`), бесплатный тир, без комиссии OpenRouter.

2026-07-19: `nano` и `flash` переведены на `gemini_direct` (самые частые по объёму
уровни — тут основная экономия). `pro` (Claude Sonnet) и эмбеддинги (OpenAI) остались
на OpenRouter — бесплатный Gemini API их не покрывает, а объём вызовов там ниже.

## Установка (Windows)

```
cd core\router_ai
python -m venv venv
venv\Scripts\pip install -r requirements.txt
copy .env.example .env
```

Вставь в `.env`:
- `OPENROUTER_API_KEY` — ключ уже был передан и вставлен при первой настройке; если это
  свежая копия проекта, ключ свой, чужой не наследуется (см. `../../SETUP_NEW_USER.md`).
- `GEMINI_API_KEY` — корпоративный ключ Gemini (Google AI Studio → Get API key,
  привязан к GCP-проекту `serviceAI`). Нужен только уровням с `provider: gemini_direct`
  в `models.yaml` (сейчас nano и flash). Без биллинга — работает в рамках бесплатной
  квоты запросов/минуту; если упрётесь в лимит на реальной нагрузке, `_chat_gemini_direct`
  в `client.py` уже делает retry с паузой на 429, но постоянные 429 — сигнал, что
  бесплатного тира не хватает и нужен либо биллинг, либо возврат уровня на OpenRouter.

## Проверка

```
venv\Scripts\python healthcheck.py
```

Печатает `OK ...` с остатком кредитов на ключе и результатом одного тестового
вызова (несколько токенов, стоимость пренебрежимо мала), либо `FAIL ...` с причиной.
Первая строка — `GET /api/v1/key`, бесплатный запрос, токены не тратит.

Чтобы проверка попадала в общий `python check_connections.py` — добавлена в его
список (`core/router_ai`).

## Использование

```python
from client import RouterAIClient

client = RouterAIClient()

# классификация (уровень nano, дёшево)
topic = client.classify("Кофемашина не наливает молоко на точке...", categories=["оборудование", "оплата", "доступы"])

# суммаризация (уровень flash по умолчанию)
summary = client.summarize(text_блок_сообщений_discord, instructions="Выдели инциденты и открытые вопросы")

# сводный отчёт с обязательными ссылками на источники — см. core/reports/
answer = client.report(question="Что произошло за неделю?", items=[...])

# эмбеддинги для базы знаний
vectors = client.embed(["текст чанка 1", "текст чанка 2"])
```

## Затраты

Каждый вызов пишет строку в `../../logs/router_ai_usage.csv`: время, метка задачи
(`task_label`), уровень, **провайдер**, модель, токены, стоимость (0 у gemini_direct —
бесплатный тир), время ответа. Файл не в git (`logs/` в `.gitignore`) — это реальные
операционные данные использования, не код.

Старый лог (до перехода на gemini_direct) без колонки `provider` — при первом же
запуске `client.py` после обновления `_ensure_usage_log_schema()` сам домигрирует
файл (проставит `provider=openrouter` всем старым строкам), ничего делать вручную
не нужно.

Разбивка "по провайдеру" (и грубая оценка сэкономленного на переходе на бесплатный
Gemini) видна на странице `/usage` веб-приложения (`core/webapp`, `usage_stats.py`).

Бюджет OpenRouter на старт — $10 (для проверки работоспособности; теперь тратится
только на pro-уровень и эмбеддинги, основной объём ушёл на gemini_direct). Если этого
мало для регулярной работы — пополнить кредиты на openrouter.ai/settings/credits,
конфиг менять не нужно.

## Переход на локальную LLM в будущем

Если/когда часть задач переведут на локальную модель — меняется только `.env`
(`OPENROUTER_BASE_URL` на адрес локального OpenAI-совместимого сервера, например
Ollama/vLLM/LM Studio) и, при необходимости, имена моделей в `models.yaml`. Код
`client.py` не трогается — он уже написан под OpenAI-совместимый формат запроса,
которому следует и OpenRouter, и большинство локальных серверов.
