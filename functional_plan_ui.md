# Функциональный план: три экрана поверх готовых коннекторов

Дополняет [architecture_plan.md](architecture_plan.md) — там целевая архитектура целиком,
здесь — детализация трёх конкретных экранов, которые запросил пользователь, плюс выбор
моделей OpenRouter под каждую задачу. Статус коннекторов и текущий прогресс — в
[PROGRESS.md](PROGRESS.md).

## 0. Контекст

Все источники данных уже подключены и рабочие: Tracker (DrinkX), Service Desk (DrinkX),
Fibbee ERP, Google Drive, Discord, Gmail. Notion пока заблокирован (ждём права). Каждый
коннектор пишет в свою SQLite с полем `raw_json` — это уже готовая сырая база для трёх
экранов ниже. Общего backend/API/UI/базы знаний ещё нет — это следующий этап по
`architecture_plan.md` (шаги 1 и 6–8), и он реализуется через три модуля из этого документа.

## 1. Модуль 1 — единый список задач

### 1.1 Источники и группировка

Двухуровневая группировка, как просил пользователь:

1. **По источнику** (жёстко, не LLM):
   - **Оперативные**: Fibbee ERP (`fibbee_tickets`), Service Desk (`sd_tickets` — заявки).
   - **Стратегические**: Tracker DrinkX (`tracker_tasks`).
2. **По смыслу внутри источника** (тема/кластер) — здесь нужна классификация. Два варианта:
   - **Эмбеддинг-кластеризация** (дёшево, разово при индексации): эмбеддинг заголовка+описания
     каждой задачи → k-means/HDBSCAN → человеку показываем кластер с авто-названием (одна короткая
     LLM-генерация названия на кластер, не на задачу). Пересчёт кластеров — периодически (раз в
     день) или по кнопке, не на каждый рендер.
   - **LLM-классификация по фиксированному списку тем** (если темы известны заранее, например
     "оборудование", "оплата", "доступы", "жалобы клиента") — дешёвая модель, один вызов на
     задачу **только при первом появлении задачи** (кэш по `task_id` + хэш содержимого, чтобы не
     переклассифицировать при каждом обновлении статуса).
   - Рекомендация: начать с фиксированного списка тем (проще, предсказуемее, дешевле), перейти
     на кластеризацию, если темы окажутся не заданы заранее.

Карточка задачи: заголовок, статус, источник, тема, дата создания/обновления, ссылка на
оригинал (прямой URL в Tracker/SD/ERP по `id`, все три коннектора уже хранят исходные ID).

### 1.1a Сводные отчёты (реализовано, MVP)

Отдельный частый сценарий — не листать список, а спросить "что произошло за период" и
получить готовые выводы со ссылками на заявки/тикеты/задачи, на которых они основаны.
Реализовано в `core/reporting/` (см. его README): забирает данные всех трёх источников
за период → строит пронумерованный список → `router_ai_client.report()` требует от модели
ссылаться на источники как `[N]` → `generate.py` разворачивает `[N]` в реальные ссылки.
Вывод — текст в терминал (чат-режим) или HTML-файл в `reports/` для открытия в браузере.
Ссылки на карточки заявок в вебе пока не подтверждены в браузере (см. `link_templates.yaml`) —
нужно один раз сверить и поправить.

### 1.2 Обновления задач

Нужен слой отслеживания изменений — сравнение `raw_json` между двумя последовательными
`sync.py`-прогонами по каждому источнику:

- Новая таблица `task_updates(source, task_id, changed_fields, old_value, new_value, detected_at)`.
- Чистая логика diff по JSON, **без LLM** — сравнение полей быстрее и надёжнее генерации.
- Опционально: одна короткая LLM-фраза "что изменилось человеческим языком" для ленты
  обновлений (например "статус: в работе → закрыто, назначен: Иванов") — дешёвая модель,
  вызывается на уже посчитанный diff, а не на весь объект.

### 1.3 Хранение

MVP не требует немедленной миграции на Postgres: можно поднять один FastAPI-слой, который
читает все существующие SQLite (`tracker_state.json`/`.db`, `servicedesk_state.json`, Fibbee
storage) и агрегирует на лету + свою небольшую SQLite для `task_updates` и кластеров/тем.
Миграция на Postgres+pgvector — когда дойдёт очередь до общей базы знаний (модуль 2), т.к. там
вектора нужны по-настоящему.

## 2. Модуль 2 — база знаний (RAG-чат)

### 2.1 Источники

Google Drive (уже полностью извлекается, включая все листы Sheets и заметки Slides), Discord
(история сообщений), плюс сами задачи/заявки/тикеты как документы (заголовок+описание+комментарии).
Notion добавится тем же пайплайном, когда откроют доступ.

### 2.2 Пайплайн

1. **Чанкинг**: документы Drive — по разделам/страницам (уже частично сделано, PROGRESS.md
   фиксирует полную длину текста без обрезки); сообщения Discord — по смысловым блокам
   (тред/окно из N сообщений подряд от одного треда или в пределах 15 минут).
2. **Эмбеддинги**: через OpenRouter Embeddings API (`/api/v1/embeddings`, OpenAI-совместимый
   формат) — см. §4.3 по модели. Кэш по хэшу текста чанка, пересчёт только при изменении.
3. **Хранение векторов**: pgvector (см. §5) — единственное место, где полноценная миграция с
   SQLite оправдана сразу.
4. **Retrieval**: top-k похожих чанков по вопросу пользователя (косинусное сходство).
5. **Генерация ответа**: LLM получает вопрос + найденные чанки с их источниками (файл/ссылка,
   автор+дата для Discord) → отвечает и обязан проставить ссылки на источники в ответе.

### 2.3 Цитирование источников

Промпт жёстко требует формат: каждый фактологический кусок ответа сопровождается ссылкой вида
`[источник](URL)` — для Drive это `webViewLink` файла (уже есть в метаданных), для Discord —
`discord://channel/message_id` или прямая ссылка на сообщение (`https://discord.com/channels/...`).
Если релевантного чанка нет — модель обязана сказать "не нашёл в базе знаний", а не придумывать.

## 3. Модуль 3 — Discord-хронология и мониторинг

### 3.1 Подсветка упоминаний

**Не требует LLM** — чистое сопоставление по ID ролей "monitoring team" / "service team" в
`message.mentions.roles` (или по тексту `@Monitoring`/`@Service`, если это не ролевые, а
текстовые упоминания) на этапе синхронизации Discord-коннектора. Помечаем сообщение флагом
`mentions_monitoring` / `mentions_service` прямо в БД — в UI просто фильтр/подсветка, без
рантайм-вызовов модели.

### 3.2 Обобщение за период

Пользователь выбирает период (последний час / сутки / произвольный диапазон) → берём все
сообщения за период → одна суммаризация одним вызовом (не по сообщению). Если сообщений много
(сутки в активном канале могут быть 500+), делаем map-reduce: сообщения бьются на окна
~3000–4000 токенов, каждое окно суммаризируется дешёвой моделью, затем сводится в финальную
сводку тоже дешёвой/средней моделью. Промпт явно просит отдельно выделить: инциденты/проблемы,
упоминания monitoring/service team, открытые вопросы без ответа.

### 3.3 Комплексы онлайн + тикеты ERP → дашборд

- **Статус комплексов**: `fibbee_healthchecks` уже собирается коннектором — поллинг с разумным
  интервалом (например раз в 1–5 минут через `sync.py`/cron), в UI просто таблица/карта статусов
  без LLM.
- **Тикеты ERP с переходом на дашборд**: у Fibbee страница "Дашборд" комплекса — SPA-шелл на
  WebSocket (см. PROGRESS.md, находка про WS), но все данные на ней дублируются в REST
  (`orders`/`healthchecks`/`supervisor-config`), которые уже забираются коннектором. Поэтому
  для MVP — просто **прямая ссылка на нативный дашборд ERP** (`erp.fibbee.com/.../dashboard/{id}`,
  открывается в новой вкладке, авторизация у пользователя уже есть в браузере) вместо
  переизобретения WS-интеграции. Свой дашборд из REST-данных — по желанию позже, не в MVP.

## 4. OpenRouter — выбор моделей

Цены на OpenRouter меняются часто (см. живые данные на `openrouter.ai/models` перед стартом
реализации) — здесь фиксирован принцип маршрутизации по уровням сложности, а не жёсткие цифры.

### 4.1 Принцип: 3 уровня + прямой код без LLM

| Уровень | Для чего | Класс моделей (июль 2026) | Почему |
|---|---|---|---|
| **0 — без LLM** | подсветка упоминаний, diff обновлений задач, фильтры/сортировка, деривация ссылок | — | это детерминированная логика, вызов модели тут — просто трата токенов |
| **1 — nano/lite** | классификация задачи по теме (разово, с кэшем), короткие пофразовые суммаризации Discord-окон в map-стадии, генерация "что изменилось" фразы | `google/gemini-2.5-flash-lite`, `openai/gpt-5-nano` (или `gpt-5-mini`) | на порядок дешевле, задача простая и структурированная — не нужна сильная модель |
| **2 — flash/haiku** | обычные ответы RAG-чата (вопрос+3–5 найденных чанков), финальная сводка Discord (reduce-стадия) | `google/gemini-2.5-flash`, `anthropic/claude-haiku-4.5` | баланс цена/качество, справляется с синтезом по нескольким источникам |
| **3 — sonnet/pro (по эскалации)** | сложные вопросы к базе знаний, требующие рассуждения по многим документам сразу; координатор-агент в будущем | `anthropic/claude-sonnet-4.6`, `openai/gpt-5` | вызывается только когда модель уровня 2 сама сигнализирует низкую уверенность, или пользователь явно просит "разберись подробно" |

### 4.2 Эмбеддинги

`openai/text-embedding-3-small` через `/api/v1/embeddings` (OpenAI-совместимый эндпоинт
OpenRouter) — как база: дёшево, проверенное качество, единый размер вектора для всей базы
знаний. Если объём вырастет и цена станет ощутимой — рассмотреть `qwen/qwen3-embedding-0.6b`
(дешевле, открытая модель), но тогда придётся пересчитать все эмбеддинги заново — модели с
разным размером/пространством векторов между собой несовместимы, менять на середине проекта
дорого. Лучше сразу решить и не менять.

### 4.3 Экономия токенов — конкретные приёмы

- **Кэш по content-hash**: эмбеддинги и LLM-классификация не пересчитываются, если текст не
  изменился (хэш чанка/задачи как ключ).
- **Не суммаризировать по одному сообщению** — только батчами/окнами.
- **RAG вместо "залить всё в контекст"**: чат по базе знаний получает только top-k релевантных
  чанков, а не все документы разом — это единственный способ удержать стоимость вопроса
  предсказуемой независимо от размера базы.
- **Маршрутизация по сложности вопроса**: короткий фактологический вопрос → уровень 1–2 сразу;
  эскалация на уровень 3 — по явному запросу или низкой уверенности модели уровня 2 (просим
  модель вернуть поле `confidence` в структурированном ответе).
- **Единый `router_ai_client`** (см. architecture_plan.md §5) — логирует токены и стоимость на
  каждый вызов с меткой задачи (`classification`, `embedding`, `rag_answer`, `summary`), чтобы
  через неделю реальной работы было видно, где деньги реально уходят, и перенастроить уровни.

## 5. Расширение общей архитектуры данных

Соответствует `architecture_plan.md` §3, но с конкретикой под три модуля:

```
SQLite коннекторов (tracker/SD/fibbee/discord/drive, уже есть)
        │  читаются как есть на старте, без миграции
        ▼
FastAPI read-layer (новый) ── task_updates.db (diff-слой, новая, SQLite ок)
        │
        ▼
Postgres + pgvector (новая, только для базы знаний/модуль 2)
        │
        ▼
router_ai_client (новый, единая точка входа в OpenRouter, 3 уровня моделей + эмбеддинги)
        │
        ▼
Web UI: 3 экрана (задачи / чат по базе знаний / Discord+мониторинг)
```

## 6. Поэтапный план реализации

1. **`router_ai_client`** (готово, `core/router_ai/`) — модуль с конфигом уровней моделей
   (`models.yaml`/`.env`), логированием токенов/стоимости в `logs/router_ai_usage.csv`,
   единым интерфейсом `chat() / classify() / embed() / summarize() / report()`. Проверка —
   `core/router_ai/healthcheck.py`, встроена в общий `check_connections.py`.
   Бюджет на старт — $10 (тестовый), возможен переход на локальную LLM позже — меняется
   только `.env` (`OPENROUTER_BASE_URL`), код клиента не трогается.
1a. **Сводные отчёты по заявкам** (готово, MVP, `core/reporting/`) — раньше остальных
    модулей 1–3, по отдельному запросу: см. §1.1a выше.
1b. **Веб-интерфейс** (готово, MVP, `core/webapp/`, тёмная тема, FastAPI+HTMX, без
    Node-сборки) — по запросу пользователя реализован сразу для всех модулей 1–3 разом,
    а не по одному, вопреки исходному порядку в этом плане. Список упрощений v1 —
    `core/webapp/README.md` (главное: группировка задач по темам ещё не сделана, "обновления"
    — сортировка по updated_at, а не полноценный diff-слой, статус комплексов — по последней
    синхронизации, не живой поллинг, подсветка Discord — по ключевым словам, не по ID ролей).
    База знаний (модуль 2) реализована без Postgres/pgvector — брутфорс-косинус по
    SQLite-кэшу эмбеддингов, осознанно (см. `core/webapp/kb_index.py`): на объёме данных
    этого проекта (634 файла Drive, ~30К сообщений Discord) избыточно поднимать Postgres
    ради этого; переезд на pgvector остаётся опцией, если объём вырастет на порядок.
2. **FastAPI read-layer** поверх существующих SQLite + `task_updates.db` (diff-логика) —
   без этого не заработает ни один из трёх экранов.
3. **Модуль 1 (список задач)**: группировка по источнику, классификация по темам (уровень 1,
   с кэшем), лента обновлений, ссылки на оригиналы. Полностью независим от базы знаний — можно
   сдать первым.
4. **Модуль 3 (Discord + мониторинг)**: подсветка упоминаний (без LLM), обобщение за период
   (map-reduce, уровни 1→2), статус комплексов онлайн, тикеты ERP со ссылками на дашборд.
   Тоже независим от модуля 2.
5. **Модуль 2 (база знаний, RAG-чат)**: самый тяжёлый — миграция на Postgres+pgvector,
   индексация Drive+Discord+задач, retrieval, генерация ответа с цитированием. Делать последним,
   т.к. зависит от стабильного индекса всех источников.
6. **Полировка**: единый веб-UI на три экрана, обзор затрат на LLM (дашборд по логам
   `router_ai_client`), подключение Notion в модуль 2, когда откроют доступ.

Дальше — координатор-агент с function-calling (фаза 6 из `architecture_plan.md`) уже поверх
готовых модулей 1–3, отдельным этапом.

## 7. Что нужно от пользователя перед стартом

- Ключ API OpenRouter (аккаунт уже подготовлен) + оценка бюджета в токенах/деньгах на месяц —
  чтобы выставить разумные значения уровней 1–3 в конфиге, не наугад.
- ID ролей Discord "monitoring team" и "service team" (или точный текст, если это не ролевые
  упоминания) — нужно для подсветки в модуле 3.
- Список тем для классификации задач в модуле 1 (если он есть готовый в компании) — иначе
  начнём с эмбеддинг-кластеризации и уточним темы по факту.
