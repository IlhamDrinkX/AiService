# Google Drive connector

Read-only. Сканирует My Drive и все Shared Drives, к которым у аккаунта есть доступ — без прав администратора Workspace, потому что используется обычный OAuth того же пользователя.

Извлекает текст из: Google Docs (целиком), Google Sheets (**все листы**, через Sheets API), Google Slides (видимый текст **и заметки докладчика**, через Slides API), PDF, .docx, .xlsx, .pptx, .rtf, .txt, .csv. Остальные типы (картинки, архивы, бинарники) — только метаданные, без текста; картинки/фото уйдут в отдельный этап парсинга фото по общему плану.

## Настройка Google Cloud

Можно использовать тот же проект и тот же OAuth-клиент, что уже настроен для Gmail:

1. В том же проекте (`sheets-495712`) включи **Google Drive API**, **Google Sheets API** и **Google Slides API** (APIs & Services → Library → найти каждый → Enable). Все три обязательны — без Sheets/Slides API коннектор упадёт с 403 при попытке прочитать таблицу/презентацию.
2. На вкладке **Data Access** добавь scopes: `drive.readonly`, `spreadsheets.readonly`, `presentations.readonly`.
3. Скопируй `credentials.json` из `connectors/gmail/` в `connectors/drive/` (клиент тот же, просто новые scopes запросятся при входе).

### Если коннектор уже был настроен раньше (до 2026-07-19)

Токен, сохранённый до добавления Sheets/Slides API, не покрывает новые scopes — `auth.py` детектирует это автоматически (`_needs_reauth()`) и откроет браузер для повторного согласия при следующем запуске `sync.py`. Дополнительно один раз запусти миграцию, чтобы уже засинканные Sheets/Slides были перечитаны полностью (см. ниже «Если коннектор уже запускался раньше»).

## Запуск

```bash
cd connectors/drive
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
cp .env.example .env
python sync.py
```

Первый запуск — браузер, вход, согласие на read-only доступ к Drive (отдельное окно от Gmail, т.к. другой scope). Дальше можно запускать `python sync.py` периодически — при повторных запусках текст перечитывается только для изменившихся файлов (сравнение по `modifiedTime`), остальное — быстрый обход метаданных.

## Проверка результата

```bash
sqlite3 data/drive.db "SELECT drive_name, name, mime_type FROM drive_files LIMIT 10;"
sqlite3 data/drive.db "SELECT COUNT(*) FROM drive_files;"
sqlite3 data/drive.db "SELECT COUNT(*) FROM drive_files WHERE text_content IS NOT NULL;"
```

## Длина извлекаемого текста

По умолчанию (`TEXT_MAX_CHARS=0`) документы читаются **целиком**, сколько бы
в них ни было страниц — PDF/Docs/docx/xlsx/txt разбираются полностью, без
обрезания. Раньше дефолт был `TEXT_MAX_CHARS=50000` (~15-20 страниц), из-за
чего длинные документы молча обрезались — это исправлено (2026-07-19).
`TEXT_MAX_CHARS` остался в `.env` как опциональный предохранитель, если
осознанно нужно ограничить размер текста (например, для контроля контекста
при последующей обработке ИИ) — ставить ненулевое значение вручную.

Если коннектор уже запускался раньше со старым лимитом, уже сохранённые
длинные документы останутся обрезанными сами по себе — `sync.py`
перечитывает файл только при изменении `modifiedTime` в Drive, а у уже
обрезанных документов он не менялся. Один раз после обновления запусти:

```bash
python reset_truncated.py   # помечает ранее обрезанные файлы на переизвлечение
python sync.py               # перечитывает их текст уже целиком
```

Аналогично: если коннектор синкал Google Sheets/Slides ДО 2026-07-19 (когда Sheets читались только первым листом, а Slides — без заметок), запусти один раз:

```bash
pip install -r requirements.txt   # добавились python-pptx, striprtf
python reset_sheets_slides.py     # помечает все Sheets/Slides файлы на переизвлечение
python sync.py                    # первый запуск откроет браузер для согласия на новые scopes,
                                   # дальше перечитает все листы / текст+заметки целиком
```

## Ограничения этой версии

- Полное сканирование метаданных на каждый запуск (не Changes API) — нормально до нескольких десятков тысяч файлов, дальше стоит перейти на инкрементальный Changes API.
- Удалённые/перемещённые в корзину файлы не подчищаются из локальной базы автоматически.
- `files.export` (используется для Google Docs) имеет собственный потолок Google — экспортируемый файл не может быть больше 10 МБ; это ограничение самого API, обойти его на уровне коннектора нельзя. Sheets/Slides этого лимита больше не касаются, т.к. читаются через свои API, а не через export.
- OpenDocument-форматы (`.odt`/`.ods`) пока не поддержаны (нет своего парсера в зависимостях) — известный, но не реализованный пробел; добавить можно через `odfpy`, если понадобится.
