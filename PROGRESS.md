# Прогресс проекта

Три вещи: что задумано изначально, что реально готово сейчас, что дальше.
Полная целевая архитектура (FastAPI, Postgres+pgvector, Celery, Router AI,
координатор-агент, веб-UI) — в [architecture_plan.md](architecture_plan.md).
Этот файл — короткий срез "где мы сейчас" по факту, без переписывания
исходного плана.

## План (кратко)

Изначальный план (architecture_plan.md) описывает целевую систему целиком:
общий backend на FastAPI, Postgres+pgvector как БД и база знаний, Celery для
фоновых задач, координатор-агент с function-calling через Router AI, веб-UI.

На практике решили строить снизу вверх: сначала довести до рабочего состояния
каждый источник данных как независимый модуль (свой venv, свой SQLite,
свой `healthcheck.py`), и только потом собирать их в общий backend/UI/агента.
Это осознанное отступление от исходного плана, а не забытая часть — модули
спроектированы так, что их SQLite-схемы легко переносятся в Postgres, когда
дойдёт очередь до общей инфраструктуры.

### Service Desk: добавлена запись (2026-07-19)

По просьбе пользователя коннектор расширен с read-only до полноценного
read+write (создание заявок, смена статуса, назначение инженера, действия
L1, диагностика, комментарий инженера, загрузка файлов) — детали и примеры
кода в `connectors/servicedesk/README.md`.

Как искали write-эндпоинты: открытого API нет и документации нет, поэтому
единственный способ — перехватить реальный трафик SPA во время живых
действий. Claude in Chrome (стандартное расширение) не подключился;
Control_Chrome (альтернативный коннектор, который подключил пользователь)
оказался написан под macOS (AppleScript) и не заработал на Windows. Рабочий
путь: пользователь подключил Desktop Commander (терминал на его реальной
машине) → закрыли Chrome → перезапустили с `--remote-debugging-port` на том
же профиле (сессия SD сохранилась) → Playwright подключился по CDP и либо
сам вёл браузер (клики/заполнение форм), либо слушал трафик, пока действия
выполнялись руками. Опрос "обнови F5 сам" не сработал (перехват событий не
поймал ручной клик пользователя вовремя) — рабочим оказался вариант, когда
скрипт сам дергает `page.reload()`/`page.click()` и слушает события в том же
вызове.

Ключевая находка про API: `PUT /api/prototype/tickets/{id}` — единственный
write-эндпоинт, full-replace (не patch), и **код заявки (`SD-xxxx`)
присваивает не сервер, а фронтенд** (сервер хранит что прислали). Первая
тестовая заявка была создана без этой логики и осталась без кода в проде —
пофиксили в `client._next_ticket_code()` и перепроверили. Все методы
проверены end-to-end через чистый `curl_cffi` (без браузера) на выделенном
тестовом клиенте/объекте `цццццц`/Day2Day/5-ка Бауманская, который указал
пользователь. В проде остались тестовые заявки с пометкой "ТЕСТ" в
заголовке (одна без кода, SD-1027, SD-1028) — DELETE-эндпоинта нет, удалять
не пробовали, не входило в задачу.

### Service Desk: запись расширена на все справочники (2026-07-19, тот же день)

По следующей просьбе пользователя запись расширена с "только заявки" до
полного набора сущностей: клиенты, объекты, комплексы, сотрудники,
проблемы, справочники (виды работ / причины отказов / узлы), плюс привязка
проблемы к заявке и запрос заявок за диапазон дат. Тот же приём discovery
(Desktop Commander + Chrome с `--remote-debugging-port` + Playwright), уже
работал без сюрпризов со второй попытки.

Главная находка: в API **два разных write-паттерна**, не один, и они не
угадываются по аналогии — пришлось проверять каждую сущность отдельно:
- tickets/objects/complexes/employees/problems/dictionary-groups —
  "prototype"-паттерн: `PUT /api/prototype/<collection>/{id}`, тело
  `{"payload": {...полный объект...}}`, id генерирует клиент, один и тот же
  запрос и создаёт, и обновляет.
- clients — обычный REST: `POST /api/clients` (без обёртки, сервер сам
  назначает id) + `PUT /api/clients/{id}` (тоже без обёртки). Если бы
  предположили по аналогии с tickets — получили бы 400 или странный id.

Другие находки: `dictionary-groups` меняются НА УРОВНЕ ГРУППЫ целиком (id в
пути — id группы вроде "work-types", а не элемента справочника; добавить
одно значение = переслать весь items[] с довеском). Привязка проблемы к
заявке — не отдельный эндпоинт, а обычное поле `ticket.problemId`. Клиенты
требуют `contractDate`/`contractUntil` как обязательные ISO-даты — пустая
строка даёт `400 Bad Request`, не сразу было очевидно из формы в UI (там
дата предзаполнена по умолчанию). Заметили гонку при двух очень быстрых PUT
подряд к одной записи справочника — подробности и как её обходить в
`connectors/servicedesk/README.md`.

Все новые методы проверены end-to-end через `curl_cffi` (без браузера) на
тестовых записях с пометкой "ТЕСТ"/"discovery" в названии — полный список
что осталось в проде см. в README коннектора.

### Fibbee ERP: полноценный коннектор вместо одного read-only эндпоинта (2026-07-19)

По просьбе пользователя ("реализовать полный функционал erp.fibbee.com,
протестировать через браузер: состояния комплексов, дашборды, история
заказов, тикеты, логи заказов для анализа, дать максимально полный
контроль агенту") коннектор расширен с единственного read-only метода
(`get_sales_points`) до полноценного клиента.

Discovery — тот же приём, что и для Tracker/Service Desk
([[feedback-endpoint-discovery-technique]]): Desktop Commander → Chrome с
`--remote-debugging-port` (для этого сайта — на отдельном свежем профиле,
свежий Chrome не разрешает remote-debugging на дефолтном профиле
пользователя) → Playwright по CDP → статический разбор фронтенд-бандла.
Бандл оказался неминифицированным по строкам — весь API-клиент нашёлся
одним проходом: **125 эндпоинтов** одной командой grep по `action:"/`.
Каждый нужный эндпоинт затем подтверждён живым запросом через `curl_cffi`.

Что добавлено в `connectors/fibbee/client.py`: `get_orders`/`get_order`
(история заказов + поле `productDump` — это и есть "лог заказа" для
анализа, отдельного эндпоинта под логи не нужно), `get_tickets`,
`get_changes_log`, `get_incidents`, `get_sales_point_healthchecks`,
`get_supervisor_config`, `get_dashboard_html`, плюс write-методы
(`create_ticket`, `update_ticket`, `update_sales_point`,
`update_order_review`, `refund_order`, `supervisor_device_up/down`,
`supervisor_startup/shutdown`) — реализованы, но **не вызывались на боевых
данных**: тикеты дублируются в Discord команды поддержки
(`discordLink`/`slackUrl`-вебхук), `orders/refund` — реальные деньги,
`supervisor/*` — прямое управление физическим оборудованием на боевой
точке. Прежде чем звать эти методы на реальном `salesPointId` — спросить
пользователя.

Две находки, которые стоило бы знать заранее (записаны в
`connectors/fibbee/README.md` подробно):
1. **Не все параметры-фильтры работают одинаково между похожими
   list-эндпоинтами.** `orders/list` реально фильтрует по
   `startDate`/`endDate`; `tickets/list` и `changes-log/list` эти же
   параметры молча игнорируют (проверено контрольным сравнением: разные
   диапазоны дают идентичный результат). `sync.py` для этих двух листает
   страницы и останавливается сам по убыванию даты в ответе — с жёстким
   потолком страниц на случай сбоя допущения о сортировке. Первая версия
   `sync.py` без этой поправки зависла в бесконечной пагинации тикетов —
   поймали вручную по логу, а не по факту зависания процесса.
2. **Ссылка "Дашборд" на карточке комплекса — не то, чем кажется при
   перехвате трафика.** Похоже на сервер-рендеренный HTML со всеми
   данными (заказы, состояние оборудования) в одном документе — при
   перехвате через Playwright переключение вкладок не порождало новых
   `fetch`/`xhr`/`document`-запросов. На деле это SPA-шелл, данные грузятся
   по **WebSocket**, который не виден в перехвате fetch/xhr/document (тот
   же класс проблемы, что SSE в п.9 памяти про discovery, но для WS).
   Прямой GET без браузера отдаёт ~360 байт пустого шелла. Хорошая новость:
   всё, что показывает эта страница, дублируется в чистых REST-эндпоинтах
   (orders/healthchecks/supervisor-config) — WebSocket не нужен для
   мониторинга, только для live-потоковой телеметрии оборудования (не
   реализовано, отдельная задача при необходимости).

Схема SQLite расширена (`storage.py`): `fibbee_orders` (с колонкой
`product_dump`), `fibbee_tickets`, `fibbee_changes_log`,
`fibbee_healthchecks` — везде `raw_json` с полным объектом, как и раньше
для `fibbee_sales_points`. Боевой прогон `sync.py` (окно 2 дня, все ~95
комплексов): 95 комплексов, 2221 заказ, 3736 тикетов, 16415 записей
аудит-лога — данные проверены (пример заказа с `productDump`, пример
тикета с `discordLink`).

### Google Drive: длинные документы больше не обрезались (2026-07-19)

По просьбе пользователя ("должен прочитывать полностью документ, даже если
в нем больше 50 страниц") нашли и поправили баг: `TEXT_MAX_CHARS=50000` был
дефолтом с самого начала, а `[:max_chars]` резался безусловно после полного
извлечения (`extract.py`) — то есть PDF/Docs/docx/xlsx честно разбирались
целиком, но результат тут же обрезался до ~15-20 страниц. Сам цикл
извлечения по каждому типу файла (`for page in reader.pages` и т.п.)
проблем не имел — упирался только финальный срез.

Фикс: `TEXT_MAX_CHARS` теперь `0` по умолчанию (без ограничения),
опционально можно поставить вручную, если осознанно нужно урезать текст.
`_cap()` в `extract.py` дополнительно логирует, когда реально что-то режет
— раньше обрезание было полностью тихим.

Важный нюанс: `sync.py` перечитывает файл только при изменении
`modifiedTime` в Drive, поэтому уже обрезанные раньше документы сам себя
не почистили бы. Добавлен `connectors/drive/reset_truncated.py` —
одноразовая миграция, которая находит файлы с `text_content` ровно
`TEXT_MAX_CHARS` (старый лимит) символов и обнуляет им `modified_time`,
форсируя `sync.py` перечитать заново. Прогнали на боевой базе (9600
файлов, 674 с текстом): нашлось 42 обрезанных файла, после
`reset_truncated.py` + `sync.py` все переизвлечены целиком — самый большой
теперь 472385 символов (был обрезан до 50000). Подробности — в
`connectors/drive/README.md`, раздел "Длина извлекаемого текста".

### Google Drive: максимум данных — все листы Sheets, заметки Slides, +pptx/rtf (2026-07-19)

По просьбе пользователя ("Google Таблицы должны экспортироваться всеми
листами", "собирать максимум данных с гугл драйва и документов") закрыли
ранее задокументированный пробел и расширили покрытие типов файлов:

- **Google Sheets**: раньше экспорт шёл через `files.export(text/csv)`,
  что физически может отдать только первый лист. Теперь читается через
  Sheets API (`spreadsheets.get` за списком листов + `spreadsheets.values.get`
  на каждый) — все вкладки, с разделителем `# Sheet: <название>`.
- **Google Slides**: раньше `files.export(text/plain)` отдавал только
  видимый на слайде текст. Теперь читается через Slides API
  (`presentations.get`) — видимый текст **и заметки докладчика**.
- Добавлены **.pptx** (`python-pptx`) и **.rtf** (`striprtf`) — раньше
  молча пропускались как неподдерживаемые типы.
- Google Docs не тронуты — `files.export(text/plain)` для них и так отдаёт
  документ целиком, доп. API не нужен.

Потребовало новых OAuth-scope (`spreadsheets.readonly`,
`presentations.readonly`) и включения Sheets API + Slides API в Google
Cloud — `auth.py` детектирует токен со старыми scope и форсирует новый
consent (`_needs_reauth()`). Добавлен `reset_sheets_slides.py` (аналог
`reset_truncated.py`) — форсирует переизвлечение всех ранее засинканных
Sheets/Slides файлов, т.к. `sync.py` иначе пропустил бы их по
неизменившемуся `modifiedTime`.

Прогнали end-to-end на боевой базе (9600 файлов): 23 файла Sheets/Slides
помечены на переизвлечение, sync.py прошёл все файлы без ошибок scope,
самая крупная таблица ("Журнал обслуживания комплексов") — теперь **17
листов, 757600 символов** (раньше — один лист). Presentations/pptx/rtf в
текущей базе пользователя не встретились, код проверен компиляцией
(`py_compile`) на всех изменённых файлах.

Известный, сознательно не тронутый пробел: OpenDocument (`.odt`/`.ods`) —
не запрашивали, отдельная библиотека (`odfpy`) не добавлена.

Пока не начато (из исходного плана): общий backend/API, Postgres+pgvector,
Celery, Router AI, координатор-агент, веб-UI, парсинг фото.

### Router AI + сводные отчёты — первая реализация (2026-07-19)

По просьбе пользователя (передан ключ OpenRouter, бюджет $10 на тест) реализованы
первые два кода-модуля из плана выше:

- **`core/router_ai/`** — единая точка выхода в OpenRouter (`client.py`), конфиг трёх
  уровней моделей в `models.yaml` (nano=`google/gemini-2.5-flash-lite`,
  flash=`google/gemini-2.5-flash`, pro=`anthropic/claude-sonnet-4.6`), эмбеддинги
  `openai/text-embedding-3-small`. Ключ лежит в `core/router_ai/.env` (гитигнорится).
  Логирование токенов/стоимости каждого вызова — `logs/router_ai_usage.csv`.
  `healthcheck.py` встроен в общий `check_connections.py` как core-сервис (не источник
  данных, поэтому отдельный список `CORE_SERVICES` в `check_connections.py`, но тот же
  контракт OK/FAIL).
- **`core/reporting/`** — сводные отчёты по заявкам/тикетам/задачам по запросу
  пользователя ("в заявках нужны сводные отчёты... со ссылками на тикеты, на основе
  которых делаются выводы"). Читает SQLite всех трёх источников напрямую (не импортирует
  коннекторы — у них разные несовместимые зависимости в отдельных venv), строит
  пронумерованный список источников, модель обязана цитировать `[N]`, `generate.py`
  разворачивает это в реальные ссылки. Вывод — текст в терминал или HTML-файл в `reports/`.

**Не проверено end-to-end**: песочница, в которой я работаю, блокирует исходящие
запросы к `openrouter.ai` (proxy 403 `blocked-by-allowlist`) — код скомпилирован
(`py_compile`) и загрузка данных из всех трёх SQLite проверена на реальных базах
пользователя (fibbee: 500 тикетов прочитано, tracker: 500 задач, servicedesk: 0 —
таблица `servicedesk_tickets` пока пустая, `sync.py` по ней ещё не гонялся), но сам
вызов OpenRouter (`healthcheck.py`, `generate.py`) нужно прогнать на машине
пользователя, там сети нет ограничений.

**Не проверено**: `core/reporting/link_templates.yaml` — адреса карточек заявок в
вебе (`sd.drinkx.tech/tickets/{code}` и т.п.) — догадка по структуре API, не
подтверждена в браузере ни для одного из трёх источников. Пользователь на прямой
вопрос про формат ссылки SD выбрал "укажу сам", но текст не дошёл — уточнить и
поправить `link_templates.yaml` при первом реальном использовании отчётов.

### Веб-интерфейс — все три модуля разом (2026-07-19, тот же день)

Пользователь уточнил важное требование: всё должно работать локально на его ноутбуке
и управляться через веб-морду в браузере, тёмная тема. На вопрос про стек и охват v1
выбрал FastAPI+HTMX (без Node) и "все три модуля сразу" (а не по одному, как
предполагал порядок в `functional_plan_ui.md`). Реализовано в `core/webapp/`:

### Четвёртый проход: AI-анализ комплексов, "стоп всё", проверка GitHub-пуша, объяснение токенов (2026-07-19/20)

**AI-анализ состояния комплексов** (`core/webapp/complex_analysis.py`,
кнопка "Проанализировать сейчас (AI)" на /discord). Перед реализацией
посмотрели живые данные Fibbee: `product_dump` заказов оказался просто
`{menuItemId: расход}` без названий ингредиентов (бесполезен как есть для

(`json.JSONDecodeError: Unterminated string`/`Expecting value`) — вместо
того чтобы ронять весь батч, `_parse_response()` теперь при неудаче
полного `json.loads` вытаскивает отдельные валидные `{...}`-объекты по
одному через regex (наши объекты плоские, без вложенных скобок — безопасно)
и отбрасывает только оборванный хвост. Заодно уменьшили `BATCH_SIZE` до 10
и подняли `max_tokens` до 2500, попросили модель держать `reason` в
пределах 6 слов. На реальном прогоне (95 комплексов, 10 батчей): 8

### Пятый проход: реальные баги из скриншотов пользователя, коммит и пуш в GitHub (2026-07-20)

Пользователь прислал 2 скриншота с реальными проблемами:

**"Job не найден" показывался как красная ошибка при каждом заходе на
страницу.** Причина: `localStorage` браузера хранит `job_id` последнего
запроса (см. паттерн опроса в `jobs.py`) — если веб-морда перезапускалась
(а за этот день перезапускалась много раз при разработке), job из памяти
пропадает, но `job_id` в браузере остаётся, и при каждом следующем заходе
JS снова пытался его поднять и снова получал "не найдено" — бесконечно
повторяющаяся, хотя безобидная ситуация. Исправлено централизованно:
новый `_stale_job_response()` в `main.py`, который чистит сам
localStorage-ключ и показывает нейтральную (не красную) заметку — во всех
5 местах, где это могло произойти (reports/kb/discord-сводка/AI-анализ
комплексов/синхронизация Drive). Отдельно для AI-анализа комплексов
(`/discord/analyze/status`) — там `hx-swap="outerHTML"` на всю панель
целиком, поэтому "не найдено"/ошибка перерендеривают всю панель заново
(с кнопкой и таблицей), а не просто текст, иначе кнопка "Проанализировать
сейчас" пропадала бы с экрана до перезагрузки страницы.

**KB-запрос словил `RESOURCE_EXHAUSTED` (дневная квота бесплатного тира
Gemini) с сырым JSON-дампом ошибки.** Расследование нашло корневую
причину, не только симптом: `health_monitor.py` гоняет healthcheck каждого
коннектора раз в 20 минут в фоне (+ ручная кнопка "Проверить сейчас" на
/usage), а `core/router_ai/healthcheck.py` до этого делал реальный
chat()-вызов на nano — который переведён на `gemini_direct` ещё в
предыдущем проходе. 72 фоновых вызова/сутки сами по себе заметно проедали
небольшую бесплатную дневную квоту, оставляя пользователю меньше, чем он
думал, до того как он вообще успевал что-то спросить. Исправлено: новый
`client.gemini_ping()` — GET на список моделей (metadata-эндпоинт), не
считается в квоту generateContent — healthcheck.py теперь использует его
вместо реального chat-вызова, когда nano настроен на gemini_direct.
Отдельно: `_chat_gemini_direct()` теперь при `RESOURCE_EXHAUSTED`/quota
кидает понятное сообщение с реальным следующим шагом ("подожди до сброса
или переключи на OpenRouter в выпадающем списке"), а не сырой JSON.
